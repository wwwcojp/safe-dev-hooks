# ガードルール変更の設計原則

`bash_guard`/`secrets_guard` や `rules/*.json`(deny/ask/保護パスの検出ルール)を変更するときに守る原則。いずれも実装レビューをすり抜け、最終ブランチレビューで検出された失敗クラスをルール化したもの。

## 原則

1. **保護パスパターンを basename の裸置きにしない** — 裸の `settings.json` は全プロジェクトの `.vscode/settings.json` まで一致し、解除不能な deny で正当な編集を永久に止めてしまう。`.claude/` などでパススコープする(例: `*/.claude/settings.json`)。自インストール配下は解決済み絶対パスで判定する。

2. **過剰検知は安全側だが、頻出イディオムの正当操作まで止めない** — deny / write_protected の誤ブロックは一見「安全側」だが、`cat x 2>/dev/null`・`2>&1` のような日常的な読取まで止めると、deny の信頼性(deny＝本当に破壊的、という原則)が崩れる。書込保護はリダイレクト対象や変異子の引数を特定して判定し、同一セグメントの無関係トークンを巻き込まない。

3. **字句回避の封じ込めは列挙でなく境界アンカーで、かつ ReDoS 安全に** — `rm -rf /.` を `/\.{1,2}/?` のような列挙で塞ぐと `//`・`/./.` が残る。`/(?!\w)` のような固定幅の境界で網羅する。可変長の入れ子量化子を避け、既存ルールの上限(`{0,8}` 等)を踏襲する。

4. **テストは両方向を書く** — false-negative(バイパス試行)だけでなく false-positive(正当操作の誤ブロック)も必ずテストする。過剰検知の見逃しは「バイパス例しかテストしていなかった」ために残った。

5. **deny を弱める変更・deny の解除経路の追加はユーザー確認必須** — deny 層は設定で解除不可の原則(`enabled:false` 免疫を含む)を崩さない。deny 判定を緩める変更や新たな解除経路の追加は、必ずユーザーに確認する。

## 背景

これらは 0.4.0 ハードニングの最終ブランチ全体レビューで検出された C1(裸 basename)・C2(`2>/dev/null` の誤 deny)・I1(列挙の取りこぼし)、および各タスクで見つかった回避経路(refspec の src:dst 誤検知、密着リダイレクト `>file` の書込保護回避など)に対応する。詳細は `docs/superpowers/specs/2026-07-18-hooks-hardening-design.md` と `docs/security-model.md`。
