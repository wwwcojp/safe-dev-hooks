## 運用ルール(このプロジェクトの進め方)

1. **設計判断は必ずスペックの決定事項ログに記録**(理由・日付・ユーザー承認の有無)。ハードニング以降は各 spec に個別記載
2. 挙動変更はTDD(テスト先行でRED確認→実装→GREEN)。バイパス試行ケースを必ず含める
3. deny層は設定で解除不可の原則を崩さない(D5、0.4.0で `enabled:false` 免疫まで拡張)。deny判定を弱める変更はユーザー確認必須
4. 生成物(ログ・uv cache等)はコミットしない。コミット前に `git status --short` 確認
5. コミットメッセージは日本語 + `Co-Authored-By` トレーラ。**トレーラのモデル名は履歴上ゆらぎあり**(初期は `Claude Fable 5`、0.3.0/0.4.0 は `Claude Opus 4.8`)。統一方針は要決定
6. **実ユーザー名・実ホームパスを書かない**(`.claude/rules/no-personal-paths.md`)。`secrets_scan` の `real-home-path` とCIで機械検査される
7. **ドッグフーディング注意**: このリポジトリ自身のHooksを有効にしたまま `hooks/`/`rules/` を編集すると `write_protected` に遮断される。Bash経由のpythonスクリプト書込で回避するか、開発中はプラグインを無効化する(`CONTRIBUTING.md`)

## スモークテスト結果(2026-07-07 実施、履歴記録)

ローカルmarketplace経由でインストールし、非対話セッション(`claude -p`)で各Hookの実機発火を確認した(当時 v0.1.0 系、8Hook構成)。

- 導入: `claude plugin marketplace add ./` → `claude plugin install safe-dev-hooks@safe-dev-hooks`(user scope で enabled、常時トークンコスト ~0)
- 発火確認(すべて期待どおり):
  - `bash_guard`: `git push --force origin main` を deny。実行前遮断
  - `secrets_guard`: `.env` の Read を deny
  - `secrets_scan`: 書き込み内容中の AWS例示キーを検知しブロッキングフィードバック(PostToolUseのためファイル自体は作成される=既知仕様)
  - `quality_gate`: lint エラーのある .py 書き込みに ruff check 結果をフィードバック
  - `exfil_guard`: URLクエリ中の AWS例示キーを送信前に deny
  - `exfil_output_scan`: 良性の WebFetch は素通し
  - `audit_log`: `.claude/logs/audit-YYYYMMDD.jsonl` に記録
  - `notify`: 直接呼び出しでベルを返却(現在は 0.3.0 で `auto` デスクトップ通知が既定)
- 良性コマンドはすべて素通しし、誤ブロックなし
