## スモークテスト結果(2026-07-07 実施)

ローカルmarketplace経由でインストールし、非対話セッション(`claude -p`)で各Hookの実機発火を確認した。

- 導入: `claude plugin marketplace add ./` → `claude plugin install safe-dev-hooks@safe-dev-hooks`(user scope で enabled、Hooks 6イベント登録、常時トークンコスト ~0)
- ウォームアップ: 8スクリプト全てに模擬イベントを流し exit 0 を確認(README §5 の手順)
- 発火確認(すべて期待どおり):
  - `bash_guard`: `git push --force origin main` を deny(`force-push-protected`)。実行前遮断を確認
  - `secrets_guard`: `.env` の Read を deny(ルール `.env`)
  - `secrets_scan`: 書き込み内容中の AWS例示キーを検知しブロッキングフィードバック(PostToolUseのためファイル自体は作成される=既知の仕様)
  - `quality_gate`: pyproject.toml + ruff 存在下で lint エラーのある .py 書き込みに ruff check 結果をフィードバック(F401/E702検出)
  - `exfil_guard`: URLクエリ中の AWS例示キーを送信前に deny(`credentials:aws-access-key`)
  - `exfil_output_scan`: 良性の WebFetch(example.com)は素通し。deny時は PostToolUse が記録されないことも監査ログで確認
  - `audit_log`: `.claude/logs/audit-YYYYMMDD.jsonl` に SessionStart / PreToolUse / PostToolUse が記録される
  - `notify`: 直接呼び出しで `{"terminalSequence": ""}`(ベル)を返却(非対話モードでは Notification イベント自体が発生しないため直接確認)
- 良性コマンド(`echo` / 通常のWebFetch)はすべて素通しし、誤ブロックなし
- テストは scratchpad 上の使い捨てリポジトリで実施。本リポジトリの作業ツリーは clean のまま
